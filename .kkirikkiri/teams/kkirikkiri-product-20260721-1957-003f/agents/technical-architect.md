## 도메인 스택 / 메서드
| 상황 | 도구·메서드 | 이유 |
|------|------------|------|
| 게임 상태 관리 | 그리드를 2차원 배열이 아닌 "점유 셀 Set + 차량 리스트"로 표현, React 상태관리(Zustand) 또는 Vue(Pinia) | 순수 함수로 이동 가능 여부를 O(차량 길이) 판정 가능, 프레임워크 무관 로직 재사용 |

| 3D 탑뷰 렌더링 | Three.js/React Three Fiber, 직교(Orthographic) 카메라를 보드 정중앙 위에서 수직으로 배치 | glTF 모델을 그대로 활용하면서 원근 왜곡 없는 격자 정렬 뷰 확보 — Kenney Car Kit이 3D 전용(2D 스프라이트 미포함)이므로 필수 |
| 대안: 2D 렌더링 | Kenney 3D 모델을 Blender로 탑다운 오소그래픽 스냅샷 → PNG 스프라이트 사전 렌더링, Canvas/SVG로 구현 | 3D 런타임 없이 초경량 구현 가능, 모바일 성능 이점 — 단 사전 작업(렌더 파이프라인) 필요 |
| 에셋 로딩 | glTF Draco 압축 + 필요한 차량 색상/타입만 초기 로드, 나머지 lazy load | 40+ 모델 전체 로드 시 번들 크기 폭증 방지 |
| 충돌/승리 판정 | 이동 후 빨간 차의 x좌표(또는 출구 방향 좌표)가 출구 열과 일치 + 경로 셀 전부 비었는지 확인 | 단순 좌표 비교로 O(1)~O(n) 판정, 프레임마다 전체 그리드 순회 불필요 |
| 퍼즐 데이터 연동 | game-systems-designer의 JSON 스키마를 그대로 초기 상태로 로드 | 로직-데이터 계층 분리, solver 결과와 클라이언트 상태 포맷 일치 |
| 배포/저장 | Vite + React/Vue, 로컬스토리지로 진행상황, 정적 호스팅(Vercel/Netlify) | 서버 없이도 배포 가능한 클라이언트 완결형 퍼즐게임에 적합 |

## 도메인 실패 패턴
- DOM 요소의 실제 픽셀 위치를 게임 상태의 소스로 사용: 리렌더링/애니메이션 타이밍에 따라 상태와 화면이 어긋나는 동기화 버그 발생
- 이동 검증 로직을 UI 클릭/드래그 이벤트 핸들러 안에 직접 작성: 유닛테스트 불가능, 로직 변경 시 UI 코드까지 건드려야 함
- 3D 모델을 원근(Perspective) 카메라로 촬영: 격자 셀 경계가 카메라 각도에 따라 왜곡되어 정확한 그리드 정렬이 어려움 — 반드시 직교 카메라 사용
- 40개 이상 glTF 모델을 앱 시작 시 전부 프리로드: 초기 로딩 시간 폭증, 실제로는 퍼즐당 최대 16대 차량만 필요
- 애니메이션 진행 중에도 입력을 받아 다음 이동을 즉시 처리: 애니메이션 중간 상태와 논리적 게임 상태가 불일치해 차량이 겹쳐 보이는 버그
- 충돌 감지를 매 프레임 전체 6×6(또는 가변 크기) 그리드 순회로 구현: 불필요한 성능 낭비, 이동 시점에만 점유 Set으로 O(n) 검사하면 충분

## 도메인 KPI
- 이동 검증/충돌 감지/승리 조건 로직 유닛테스트 커버리지 100% (UI 없이 순수 함수 단위 테스트)
- 초기 로드 시 프리로드 3D 에셋 5종 이하 (퍼즐 시작 시 필요한 차량 색상만, 나머지 lazy load)
- 60fps 유지 (모바일 중급 기기 포함, 이동 애니메이션 중)
- 게임 로직 번들 크기 200KB 이하 (gzip, 3D 에셋 제외)
- 이동 애니메이션 150~250ms (즉각 반응 + 시각적 인지 가능 사이 균형)
