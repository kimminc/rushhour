# 러시아워(Rush Hour) 웹게임 기획 — 종합 리포트

- 팀: kkirikkiri-product-20260721-1957-003f
- 참여: 게임 시스템 디자이너(Analyst) / 기술 아키텍트(Builder) / 검증자(Critic) / 팀장(통합)
- 전체 신뢰도(Critic 판정): **중간** — 치명적 리스크 0건, 주의 등급 3건, 경미 2건

---

## 0. 사전 리서치 요약

### 러시아워 보드게임 규칙
- 보드: 6×6 격자, 지정된 변에 출구
- 차량: 자동차 12개(1×2칸), 트럭 4개(1×3칸), 빨간 차 1개(목표)
- 이동: 배치된 방향으로만 직선 슬라이드, 회전·점프 불가
- 목표: 다른 차량을 슬라이드시켜 빨간 차의 출구 경로를 확보 후 탈출
- 난이도: 카드 기반 40~155장, Beginner~Grand Master. 카드 뒷면에 최소해 인쇄
- 학술적 사실: 일반화된 Rush Hour 해결가능성 판정은 **PSPACE-complete**(Tromp & Cilibrasi, 2005). 최고난도 배치는 51회 이동 필요

### Kenney Car Kit (https://kenney.nl/assets/car-kit)
- 40개 이상 3D 차량 모델(승용차/트럭/밴), 포맷 OBJ/FBX/**glTF**(웹 표준)
- 라이선스 **CC0 1.0 Universal** — 상업적 이용 자유, 크레딧 의무 없음
- **3D 전용, 2D 스프라이트 미포함** — 이 사실이 에셋 적용 방식 논의의 전제가 됨

## 1. 게임 시스템 설계 (game-systems-designer)

### 보드 크기
**하이브리드 채택**: 이동/충돌/승리 로직과 퍼즐 데이터 스키마는 `board:{w,h,exit}` 가변 파라미터로 설계하되, 초기 UI·에셋·퍼즐 뱅크는 6×6만 제공. 상수 하드코딩을 피해 향후 확장(8×8 등) 시 렌더링·에셋만 손대면 되도록 설계.

### 난이도 시스템
- 주지표: **BFS 최소 이동수**. 보조지표: 분기계수, 강제/우회 이동수, 최소해 다중도, 차량 수
- 이동수 카운트 규약: "한 차량을 한 방향으로 이동 = 1수(칸수 무관)" — 전체 파이프라인에서 통일 필요
- 5등급: Beginner(1–5) / Easy(6–8) / Intermediate(9–12) / Advanced(13–17) / Expert(18+), 보조지표 괴리 시 재분류 플래그
- 초기 퍼즐 뱅크 57개 목표 (등급별 8~15개)

### 레벨 데이터 관리
- **오프라인/빌드타임 파이프라인**: 절차적 생성(역방향 생성 권장) → BFS solver로 검증 → 품질 필터 4단계(해 존재/비자명성/최소해 다중도 제약/대칭 중복제거) → 정적 JSON 뱅크
- **클라이언트에는 solver를 탑재하지 않음** — PSPACE-complete 특성상 브라우저에서 대량/대형 보드 탐색은 프리징 위험
- 가변 보드를 수용하는 JSON 스키마 확정 (§3 코드 참조는 원본 설계안 참고)

> **중요 — 팀장 승인 필요 항목**: 디자이너는 "유일해 100%" 요구가 슬라이딩 퍼즐 특성상 문자 그대로는 비현실적이라 판단, "solvable 100% + 비자명 + 최소해 다중도 등급별 상한"으로 재정의를 제안했습니다. Critic도 이 재정의 자체는 타당하다고 동의했으나, **제품 의도(원 KPI)를 변경하는 결정이므로 실제 승인이 필요**합니다 (권장 절충안: Advanced/Expert 등급만 다중도=1 필수).

## 2. 기술 구현 설계 (technical-architect)

### 핵심 원칙
게임 상태를 **차량 리스트 + 점유 셀 Set**으로 표현하고, 이동 검증·충돌 감지·승리 조건을 **순수 함수**로 분리 (렌더링/DOM과 완전 독립 → Vitest로 UI 없이 100% 단위 테스트 가능).

- **이동 검증**: 방향을 `+1/-1` 축값으로만 받는 API 설계로 "직선 이동만"을 타입 레벨에서 구조적으로 보장. `maxSlide()`가 진행 경로의 모든 셀(swept path)을 검사해 엔드포인트만 보는 버그를 회피
- **충돌 감지**: 정적(퍼즐 로드 시 배치 검증) + 동적(이동 시점에만 점유 Set 조회, O(진행칸수)) 2계층. 매 프레임 전수 순회 없음
- **승리 조건**: 목표차 전방부터 출구까지 경로가 완전히 비었는지로 판정. 입력은 `IDLE→ANIMATING→SOLVED` 상태머신으로 게이팅해 애니메이션 중 입력을 차단

### Kenney 에셋 적용
- **옵션 A (권장): 3D 직교(Orthographic) 카메라 탑다운.** React Three Fiber + drei, glTF+Draco. 카메라를 보드 정중앙 위에 수직 배치해 원근 왜곡 없이 격자 정렬. **런타임 머티리얼 recolor**로 모델 1개를 여러 색상 차량에 재사용(2D 방식의 "색상별 재렌더" 문제 회피)
- **옵션 B (경량 폴백): 2D 사전 렌더 스프라이트.** Blender로 탑다운 오소그래픽 스냅샷 → PNG 아틀라스 → Canvas/SVG 렌더. 3D 런타임 불필요하나 색상 변형마다 재작업 필요
- 렌더 계층을 인터페이스로 추상화해 A로 출시 후 성능 문제 실측 시 B로 드롭인 교체 가능하게 설계 권장

### 기술 스택 제안
TypeScript + Vite + React + Zustand(상태관리) + React Three Fiber/drei(렌더링) + Vitest(테스트) + localStorage(저장) + 정적 호스팅(Vercel/Netlify). 로직/상태/렌더링 3계층을 엄격히 분리.

## 3. 독립 검증 결과 (critic)

### 종합 판정
전체 신뢰도 **중간**. 치명적(구현 즉시 붕괴) 리스크는 **0건 확인**. 다만 두 전문가가 "정합을 맞췄다"고 각자 주장한 7개 계약 항목을 문자 그대로 대조한 결과, **5개는 실제 일치**했지만 **2개는 미봉합** 상태였습니다.

### 주의 등급 리스크 (3건 — 출시 전 반드시 해결)
1. **exit 슬라이드 카운트 불일치**: 디자이너의 `minMoves`는 목표차의 최종 탈출 슬라이드까지 포함해 세지만, 아키텍트의 `isSolved()`는 "경로가 빈 순간" 자동 승리로 처리해 마지막 슬라이드를 카운트하지 않습니다. 이 차이가 봉합되지 않으면 최적 플레이 시 실제 `moveCount`가 `minMoves`보다 1 적게 나오고, 저장된 해법(hint) 재생이 마지막 스텝에서 실패합니다.
   - **조치**: "경로 클리어 = 승리, 최종 슬라이드 무카운트"로 solver·isSolved·solution 데이터를 −1 기준으로 통일
2. **solution 이동 부호(+/-) 규약 미정의**: 디자이너의 데이터 스키마(`"solution":["A+1","T1-2","X+4"]`)에 `+`/`-`가 무엇을 의미하는지 명시가 없고, 아키텍트의 내부 `Axis1` 타입은 별도 규약입니다. 반대로 해석하면 힌트 기능이 차량을 반대 방향으로 밀어 불법 상태를 만듭니다.
   - **조치**: `+`= 방향축(가로는 오른쪽, 세로는 아래) 증가로 고정하는 명문 규약을 양쪽 문서에 못박고, 라운드트립 유닛테스트 추가
3. **"Rush Hour" 상표/원작 콘텐츠 리스크**: 두 설계안 모두 제품명이나 콘텐츠 출처를 다루지 않았습니다. 게임 규칙(직선 슬라이드 퍼즐) 자체는 저작권 보호 대상이 아니라 자유롭게 구현 가능하지만, ThinkFun의 등록상표 "Rush Hour"를 프로젝트명/UI에 그대로 쓰거나 정규판 40장의 정확한 차량 배치를 그대로 복제해 넣는 것은 별개의 법적 리스크입니다.
   - **조치**: 오리지널 제품명 사용(예: "슬라이드 러시" 류), 퍼즐 시드는 자체 생성분만 사용 — 규칙 구현 자체는 문제 없음

### 경미 리스크 (2건)
- 3D 60fps/번들 크기 주장이 실측 없는 가정(모바일 실시간 그림자 비용 미고려) — 실기 벤치마크 후 그림자 on/off 게이트 필요
- 애니메이션 중 입력이 차단될 때 사용자에게 아무 피드백이 없음 — 입력 버퍼링 또는 불법 이동 시 shake 애니메이션 권장

### Kenney 에셋 관련 결론
CC0 라이선스라 상업적 이용에 법적 문제는 **없음(확인함)**. 다만 3D 전용이므로 2D를 원한다면 반드시 별도 렌더 파이프라인(Blender 사전 렌더)이 선행되어야 한다는 점은 두 전문가와 검증자 모두 일치.

## 5. 결론

게임 로직(순수 함수 이동/충돌/승리 판정)과 데이터 파이프라인(오프라인 BFS solver + 품질 필터) 골격은 견고하며, Kenney Car Kit(CC0, glTF)을 3D 직교 카메라 탑다운 방식으로 적용하는 접근은 아트 비용 없이 색상 변형까지 해결하는 합리적 선택입니다. 다만 두 전문가의 산출물을 그대로 이어 붙이면 **이동수 카운트와 힌트 부호 규약이 실제로는 어긋난 채 구현이 시작될 위험**이 있었고, 이는 이번 교차 검증에서 드러났습니다. 위 "즉시 조치" 4개 항목을 확정한 뒤 구현에 들어가는 것을 권장합니다.
