# 퍼즐 절차적 생성 파이프라인

이 절차는 **오프라인(Node 스크립트 또는 빌드타임)에서만** 실행한다. solve()는 PSPACE-complete 문제의 BFS 탐색이라 브라우저에서 실행하면 프리징 위험이 있다 — Gridlock 프로젝트의 확정된 기획 결정이다.

## 1. 배치 생성
역방향 생성을 우선한다: 목표 차량이 이미 출구를 통과한 "풀린" 상태에서 시작해, 합법적인 역방향 이동을 무작위로 N회 적용한다. 순방향 무작위 배치보다 "풀리는" 비율이 훨씬 높다.

## 2. 검증
생성된 배치를 `validatePlacement()`로 정적 검증(겹침/경계/목표 정렬) 후, `solve()`로 실제 풀이 가능한지와 최소 이동수를 계산한다. 추정치를 쓰지 말고 항상 solve()의 실제 반환값을 사용한다.

## 3. 품질 필터 (4단계, 순서대로 적용)
1. **해 존재**: `result.solvable === true`가 아니면 폐기
2. **비자명성**: `result.minMoves`가 목표 등급의 하한 이상인지 확인
3. **다중도 제약** — Advanced/Expert 등급만 강제 (Beginner~Intermediate는 완화, 팀 결정): 동일 최소 이동수 해가 여러 개면 "아무 차나 밀어도 풀리는" 퍼즐이므로, 상위 등급에서는 다중도를 재계산해 상한(예: ≤3)을 넘기면 하향 조정하거나 폐기
4. **대칭 중복 제거**: 6×6·출구 고정 기준에서는 배치+차량ID 라벨을 정규화한 해시로 완전 동일 배치만 걸러낸다. 출구 위치를 자유롭게 확장하면 D4(회전4×반사2) 정규화가 별도로 필요해진다 — 그 전까지는 과설계하지 않는다.

## 4. 난이도 등급 부여
`result.minMoves` 기준: Beginner(1-5) / Easy(6-8) / Intermediate(9-12) / Advanced(13-17) / Expert(18+). 이 임계값은 실측 전 잠정치이므로, 플레이테스트 데이터가 쌓이면 재조정 대상임을 생성 리포트에 명시한다.

## 5. 출력
`PuzzleSchema` 형식(gridlock/logic/adapter.js 상단 주석 참조)으로 직렬화한다. `meta.minMoves`, `meta.difficulty`, `meta.solution`은 반드시 solve()의 실제 결과를 그대로 쓴다 — 손으로 추정한 값을 넣지 않는다. `vehicles[].type`(렌더용 모델/색상 참조)을 항상 채운다. 생성 스크립트는 트랜스파일 없이 `node generate-puzzles.js`로 바로 실행 가능하다(순수 ESM).

## 6. 생성 리포트
등급별 생성 개수, solvability 비율(생성 시도 대비 채택 비율), 평균 solve() 탐색 상태 수를 함께 보고한다. 목표 수량(등급별 8~15개, 총 57개) 미달 시 억지로 낮은 등급을 승격시키지 말고 미달 사실을 그대로 보고한다.
